# Лексическая спецификация языка Go для учебного анализатора

## 1. Лексема

Лексема — минимальная значимая единица исходного текста программы. Лексический анализатор читает текст посимвольно, пропускает пробельные символы и преобразует последовательности символов в токены.

Каждый токен хранит:

- тип лексемы;
- исходный текст лексемы;
- имя файла;
- номер строки;
- номер столбца.

Пробелы, табуляции и переводы строк не добавляются в список лексем, но учитываются при подсчёте строки и столбца.

## 2. Типы лексем

Анализатор поддерживает следующие типы:

- `Keyword` — ключевое слово;
- `Identifier` — идентификатор;
- `IntegerLiteral` — целый литерал;
- `FloatLiteral` — вещественный литерал;
- `ImaginaryLiteral` — мнимый литерал;
- `StringLiteral` — строковый литерал;
- `RuneLiteral` — rune literal;
- `Operator` — оператор;
- `Delimiter` — разделитель;
- `EndOfText` — конец текста;
- `Unknown` — неизвестная лексема.

## 3. Ключевые слова Go

Поддерживаются ключевые слова:

```text
break default func interface select
case defer go map struct
chan else goto package switch
const fallthrough if range type
continue for import return var
```

Если считанное слово совпадает с одним из этих слов, оно получает тип `Keyword`. Иначе оно считается идентификатором.

## 4. Идентификаторы

В полной спецификации Go идентификаторы могут содержать Unicode-буквы. В учебной реализации используется упрощённое правило:

- первый символ — латинская буква или `_`;
- последующие символы — латинские буквы, цифры или `_`.

Примеры корректных идентификаторов:

```text
x
main
Println
userName
_value
```

## 5. Числовые литералы

### 5.1. Целые литералы

Поддерживаются десятичные, двоичные, восьмеричные и шестнадцатеричные целые числа:

```text
123
0b1010
0B1010
0o755
0O755
0xFF
0XFF
1_000
```

Символ `_` разрешён внутри числовой записи, но не должен стоять в начале или конце числа и не должен использоваться подряд.

### 5.2. Вещественные литералы

Поддерживаются формы:

```text
3.14
10.
.25
1e10
1.5e-3
```

### 5.3. Мнимые литералы

Мнимый литерал — числовой литерал с суффиксом `i`:

```text
10i
3.14i
```

## 6. Строковые литералы

### 6.1. Обычные строки

Обычные строки заключаются в двойные кавычки:

```text
"hello"
"hello\nworld"
```

В обычных строках поддерживаются простые escape-последовательности, например `\n`, `\t`, `\\`, `\"`.

### 6.2. Сырые строки

Сырые строки заключаются в обратные кавычки:

```text
`raw string`
```

Сырая строка может занимать несколько строк. Она завершается следующей обратной кавычкой.

## 7. Rune literal

Rune literal заключается в одинарные кавычки:

```text
'a'
'\n'
'ж'
```

В учебной реализации корректно обрабатываются ASCII-символы и простые escape-последовательности. Структура метода `ScanRune()` позволяет расширить поддержку Unicode.

## 8. Операторы

Поддерживаются операторы:

```text
+ - * / % & | ^ << >> &^
+= -= *= /= %= &= |= ^= <<= >>= &^=
&& || <- ++ -- == < > = ! != <= >= := ... .
```

Многосимвольные операторы распознаются по принципу максимальной длины: сначала проверяются самые длинные варианты, затем короткие.

## 9. Разделители

Поддерживаются разделители:

```text
( ) [ ] { } , ; :
```

## 10. Комментарии

Поддерживаются два вида комментариев:

```text
// однострочный комментарий
/* многострочный комментарий */
```

Комментарии распознаются лексером, но не добавляются в список лексем и не учитываются в статистике.

## 11. Лексические ошибки

Анализатор обнаруживает следующие ошибки:

- неизвестный символ;
- незакрытая строка;
- незакрытая сырая строка;
- незакрытый rune literal;
- некорректная escape-последовательность;
- незакрытый многострочный комментарий;
- некорректная числовая константа;
- ошибка открытия файла.

Ошибки накапливаются в классе `Error`. Программа не завершает работу при первой ошибке и продолжает анализ остальных файлов.
